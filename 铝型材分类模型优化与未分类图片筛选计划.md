# 铝型材分类模型优化与未分类图片筛选计划

更新时间：2026-06-18

## 1. 当前目标

当前目标是先把 11 类铝型材缺陷分类模型做稳，并使用供货商提供的未分类图片检查模型是否存在漏检风险。

优化方向：

- 防漏检优先：宁可多报一些疑似缺陷，也尽量避免把缺陷判成正常。
- 服务器正式运行：训练、全量推理、ONNX 导出和性能测试都在服务器执行。
- 本地只做代码编写、语法检查、少量图片 smoke test 和文档维护。
- 计划不是硬性流程：后续实验顺序可以调整，核心目标只有一个，就是提高分类准确度并降低漏检风险。
- 每次修改模型或训练策略后，必须记录运行命令、模型路径、评估指标和对比结论，方便回退到更好的版本。

## 8. 分类模型优化计划

实验顺序：

1. Baseline：当前 ResNet50。
2. Class Weight：类别权重交叉熵。
3. Weighted Sampler：弱类采样增强。
4. Focal Loss：难样本优化。
5. Class Weight + Focal Loss：防漏检组合实验。
6. Lightweight：ResNet18 / MobileNetV3 速度对比。
7. ONNX：导出和服务器推理验证。

当前已经把以下训练策略写入代码，可通过 `src/classification/run_pipeline.py` 参数切换：

| 策略 | 参数 | 目的 | 优先级 |
| --- | --- | --- | --- |
| Baseline | 无额外参数 | 保留当前基准，用于对比 | 已完成 |
| Class Weight | `--class-weights` | 提高少样本类别在损失中的权重 | 优先跑 |
| Weighted Sampler | `--weighted-sampler` | 训练时更频繁抽到少样本类别 | 第二优先 |
| Focal Loss | `--loss focal --focal-gamma 2.0` | 让模型更关注难分样本 | 第三优先 |
| Class Weight + Focal Loss | `--class-weights --loss focal --focal-gamma 2.0` | 同时强化少样本和难样本 | 视前三组结果决定 |

服务器实验命令模板：


对比思路：

- 不只看 Top1 总准确率，还要看弱类是否改善，尤其是擦花、碰伤、凸粉、脏点。
- 如果 Top1 略降，但“缺陷判正常”明显减少，可以考虑保留，因为当前业务更怕漏检。
- 如果弱类提升但正常误报过多，需要降低策略强度或增加正常样本。
- 如果训练集指标很好但 raw_images 结果明显异常，优先怀疑过拟合或拍摄分布差异。
- 每次实验结束后，先用同一批 labeled 数据评估，再用 `screen_raw_images.py --no-copy` 跑 raw_images 质量基线，最后用 `compare_prediction_reports.py` 对比结果变化。

### 9.1 Class Weight 实验结论

Class Weight 已完成训练和评估。

结果判断：

- Top1 从 Baseline 的 92.45% 降到 89.59%，整体分类准确率下降。
- Top3 从 99.37% 降到 98.79%，仍然较高，但低于 Baseline。
- 缺陷判正常从 42 张降到 39 张，只减少 3 张。
- 正常误报从 31 张增加到 82 张，误报增加明显。
- 碰伤从 76.62% 提升到 85.71%，凸粉从 84.38% 提升到 100.00%。
- 擦花从 57.05% 降到 50.34%，这是当前最弱类别，下降不可接受。

当前结论：

```text
Class Weight 不作为当前首选部署模型。
可以保留为防漏检候选，但不能替换 Baseline。
下一步继续观察 Weighted Sampler 是否能在不明显牺牲 Top1 的情况下改善弱类。
```

### 9.2 Weighted Sampler 实验结论

Weighted Sampler 已完成训练和评估。

结果判断：

- Top1 从 Baseline 的 92.45% 降到 91.19%，整体准确率小幅下降。
- Top3 从 99.37% 降到 99.27%，基本接近 Baseline。
- 缺陷判正常从 42 张降到 27 张，漏检风险明显下降。
- 正常误报从 31 张增加到 71 张，误报增加较多。
- 碰伤从 76.62% 提升到 90.91%。
- 凸粉从 84.38% 提升到 96.88%。
- 脏点从 87.56% 提升到 93.78%。
- 擦花从 57.05% 降到 49.66%，仍然是最大问题。
- raw_images 中预测正常候选从 Baseline 的 442 张增加到 536 张，需要人工复核确认是否带来更多漏检。

当前结论：

```text
Weighted Sampler 是目前更有价值的防漏检候选模型。
它显著减少缺陷判正常，但正常误报增加，且擦花更差。
暂不直接替换 Baseline，继续等待 Focal Loss 和组合实验结果。
如果后续实验不能更好，Weighted Sampler 可以作为“防漏检优先模式”的候选。
```

### 9.3 Focal Loss 实验结论

Focal Loss 已完成训练和评估。

结果判断：

- Top1 从 Baseline 的 92.45% 提升到 92.59%，略高于 Baseline。
- Top3 从 99.37% 提升到 99.66%，略高于 Baseline。
- 缺陷判正常从 42 张降到 39 张，漏检小幅下降。
- 正常误报从 31 张增加到 35 张，接近 Baseline。
- 擦花从 57.05% 提升到 75.17%，这是目前最明显的收益。
- 碰伤从 76.62% 提升到 85.71%。
- 凸粉与 Baseline 持平，为 84.38%。
- 脏点从 87.56% 降到 80.86%，需要重点关注。
- raw_images 中预测正常候选从 Baseline 的 442 张增加到 554 张，人工复核压力和潜在漏检风险需要关注。

当前结论：

```text
Focal Loss 是当前最平衡的候选模型。
它在 Top1、Top3、擦花、碰伤上优于 Baseline，误报增加很小。
主要风险是脏点下降，以及 raw_images 中预测正常候选明显增加。
需要等待 Class Weight + Focal Loss 组合实验完成后，再决定最终推荐模型。
```

### 9.4 Class Weight + Focal Loss 实验结论

Class Weight + Focal Loss 已完成训练和评估。

结果判断：

- Top1 从 Baseline 的 92.45% 降到 87.94%，整体准确率明显下降。
- Top3 从 99.37% 降到 99.03%，略低于 Baseline。
- 缺陷判正常从 42 张降到 15 张，是当前漏检最低的实验。
- 正常误报从 31 张增加到 113 张，误报代价过高。
- 擦花从 57.05% 降到 42.28%，弱项进一步恶化。
- 凸粉从 84.38% 提升到 98.44%，脏点从 87.56% 提升到 97.13%。
- raw_images 中预测正常候选从 Baseline 的 442 张降到 298 张，说明模型更激进地判缺陷，但会带来大量误报。

当前结论：

```text
Class Weight + Focal Loss 不推荐作为主模型。
它适合证明“强防漏检”方向有效，但牺牲了太多整体准确率和正常误报。
如果现场极端怕漏检，可以保留为人工复核压力较大的防漏检模式候选。
```

### 9.5 当前最终推荐

截至本轮实验结束，推荐顺序如下：

1. **Focal Loss 作为当前最平衡候选模型**  
   Top1/Top3 略高于 Baseline，擦花提升明显，漏检略降，正常误报只小幅增加。

2. **Baseline 作为稳定保底模型**  
   总体稳定，raw_images 正常候选较少于 Focal，但擦花较弱。

3. **Weighted Sampler 作为防漏检候选模型**  
   漏检明显减少，但误报增加，擦花仍差。

4. **Class Weight + Focal Loss 仅作为极端防漏检参考**  
   漏检最低，但误报和 Top1 损失过大。

建议下一步：

```text
先不要直接替换上线模型。
优先下载并人工复核 Focal Loss 的 raw_images 正常候选和脏点错分样本。
如果人工复核确认 Focal Loss 没有明显新增漏检，则优先用 Focal Loss 作为下一版候选。
如果人工复核发现 Focal Loss 把缺陷判正常增加，则回退 Baseline，并继续补充擦花、脏点样本。
```

## 12. 不同机器运行结果一致性说明

同一个模型、同一批图片、同一份代码，在不同机器上运行，结果原则上应该基本一致。

但需要注意：

- CPU 和 GPU 的浮点计算可能有极小差异。
- PyTorch、TorchVision、Pillow、CUDA、cuDNN 版本不同，图片缩放和模型计算可能有细微差异。
- 如果某张图片 Top1 和 Top2 置信度非常接近，极小数值差异可能导致类别翻转。
- 因此不能只凭单张图片判断模型质量是否变化，应比较整批结果。

判断模型质量是否变化，重点看：

- `class_changed_count`：预测类别变化数量。
- `status_changed_count`：读取状态变化数量。
- `max_confidence_delta`：最大置信度差异。
- `confidence_changed_count`：超过容忍阈值的置信度变化数量。

如果：

```text
class_changed_count = 0
status_changed_count = 0
max_confidence_delta 很小
```

则认为不同机器运行结果一致，模型质量没有变化。

如果：

```text
class_changed_count 明显大于 0
```

则需要检查：

- 是否使用了同一个 `classifier.torchscript.pt`。
- 是否使用了同一个 `labels.json`。
- 图片是否完全一致。
- Python / PyTorch / TorchVision / Pillow 版本是否差异很大。
- 是否使用了不同的预处理逻辑。

```bash
  --output outputs/classification_results/raw_images_quality_compare.md
```

输出文件：

```text
outputs/classification_results/raw_images_quality_compare.md
```

推荐判定规则：

| 指标 | 判断 |
| --- | --- |
| `class_changed_count = 0` | 模型分类结果一致 |
| `class_changed_count` 很少 | 重点看变化图片是否是低置信度边界样本 |
| `class_changed_count` 很多 | 不认为一致，需要排查环境或模型文件 |
| `status_changed_count > 0` | 有图片读取状态变化，优先排查图片文件和 Pillow |
| `max_confidence_delta < 0.01` | 置信度变化很小，通常可接受 |

本阶段先写代码和比较工具，不再复制新的筛选图片。

## 15. 自动训练队列与定时唤醒

当前已启用自动训练队列：

```text
脚本：src/classification/run_experiment_queue.py
服务器队列日志：outputs/classification_results/jobs/experiment_queue.log
服务器队列 PID：3583964
当前等待的 Class Weight 训练 PID：3517135
```

队列执行顺序：

1. 等待当前 `exp_class_weight` 训练完成。
2. 对 `exp_class_weight` 自动执行：
   - 全量已分类图片评估：`full_screening/summary.md`
   - raw_images no-copy 推理：`raw_images_quality/summary.md`
   - raw_images 与 baseline 对比：`raw_images_quality_compare.md`
3. 自动训练并评估 `exp_weighted_sampler`。
4. 自动训练并评估 `exp_focal`。
5. 自动训练并评估 `exp_class_weight_focal`。
6. 汇总结果到：

```text
outputs/classification_results/experiment_queue_summary.csv
```

已启用 Codex 定时唤醒：

```text
名称：铝型材训练监控
频率：每 5 分钟
用途：自动检查服务器训练日志、进程、评估结果；有新结果时分析并更新本文档。
```

触发条件与处理规则：

- 如果队列仍在训练：只记录当前实验、PID、最新 epoch，不做额外操作。
- 如果某个实验完成：读取该实验的 `summary.md` 和对比文件，分析是否优于 Baseline。
- 如果队列失败：先看日志定位错误；能自动修复的代码问题先修复并恢复队列。
- 禁止自动删除模型权重、原始图片和已有评估结果。
- 判断是否保留模型时，优先看漏检风险，其次看弱类准确率，最后看总 Top1。
